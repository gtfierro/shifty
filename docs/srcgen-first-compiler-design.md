# Source-Generation-First Compiler Design

## Status
- Proposed
- Scope: new compiler crate + CLI wiring + shared runtime integration points
- Historical note: this document describes the migration plan that introduced srcgen; the legacy compiler crate has since been removed.

## Context
At the time this design was written, `crates/shacl-compiler` was tightly coupled to migration-era constraints and template structure. The goal was a fresh compiler context where generated Rust source is the first-class output, while preserving correctness and compatibility.

This design introduced a brand-new compiler track that was fully separate from the existing compiler crate.

## Decision
Adopt a new compiler crate, `crates/shacl-srcgen-compiler`, as a clean-room implementation with source generation as the primary concern.

- Make generated Rust validators first-class.
- Preserve semantic correctness through parity and fallback mechanisms.

## Goals
1. Source generation is the core product (quote/proc-macro2 driven modules).
2. Full separation from the old compiler crate and its internal IR/codegen constraints.
3. Deterministic, testable generated output.
4. Performance-oriented specialization while preserving SHACL semantics.
5. Controlled rollout with explicit fallback paths.

## Non-Goals
1. Immediate deletion of the old compiler during the initial migration phase.
2. Semantics changes to SHACL validation behavior.
3. JIT/VM compilation in phase 1.

## High-Level Architecture

### New Compiler Track
- Input: `ShapeIR`
- Pipeline: `ShapeIR -> SrcGenIR -> Generated Rust Modules`
- Output: generated crate modules for validation/inference/reporting

### Runtime Relationship
The generated code may call shared utilities in `lib`, but generated validators are primary execution logic. The shared runtime acts as:
- utility provider (common helpers/data structures)
- compatibility fallback for unsupported edge cases

### Migration Track
During the migration period, the old compiler remained available for:
- comparison
- fallback
- risk-managed rollout

## Proposed Module Layout

### New crate: `crates/shacl-srcgen-compiler`
- `src/lib.rs`
- `src/main.rs`
- `src/ir/mod.rs` (SrcGenIR types)
- `src/lower/mod.rs` (`ShapeIR -> SrcGenIR` lowering)
- `src/analyze/mod.rs` (compile-time analyses)
- `src/opt/mod.rs` (compile-time optimizations)
- `src/codegen/mod.rs`
- `src/codegen/modules/*.rs` (module-specific generators)
- `src/codegen/templates/*.rs` (small template snippets only)
- `src/emit/mod.rs` (file tree/materialization)

### Shared library additions in `lib`
- Optional trait boundary for generated executor integration
- Shared helper APIs only where reuse is valuable

## SrcGenIR Principles
1. Typed and explicit IR tailored for source generation.
2. Stable IDs and deterministic ordering.
3. Preserve enough structure to generate optimized validators directly.
4. Include fallback annotations for unsupported constructs.

## Generated Code Model
Generated crate should include:
1. `prelude` (imports, constants, small runtime structs)
2. `paths` (path evaluators)
3. `targets` (target resolution functions)
4. `validators_node` (node shape validators)
5. `validators_property` (property shape validators)
6. `inference` (rules execution)
7. `report` (report materialization)
8. `run` (entrypoints and orchestration)

### First-Class Requirement
Per-shape/per-component validators are generated by default for supported constructs.

### Fallback Requirement
For unsupported or high-risk constructs, generated code can delegate to shared runtime fallback APIs while maintaining output parity.

## Backend/Compiler Selection Model

### Compiler selector
During migration, the CLI exposed compiler selection between the old compiler and `crates/shacl-srcgen-compiler`.

### Backend selector (within srcgen compiler)
- `--backend specialized` (default): generated validator logic first
- `--backend tables`: bootstrap/tables mode when needed

## Compatibility and Versioning
1. Generated crate metadata includes compiler version and schema markers.
2. Compatibility tests must cover failure behavior for mismatched runtime/compiler assumptions.
3. Fallback behavior must be explicit and observable in logs/metrics.

## Testing Strategy
1. Semantic parity tests:
- `Validator` vs compiled-srcgen outputs on existing manifests, with migration-era comparison against the old compiler where needed.
2. Generated source snapshot tests:
- deterministic module output for fixed fixtures.
3. Fallback-path tests:
- force unsupported constructs and assert delegated execution behavior.
4. Performance tests:
- compare compile time, startup, and validation throughput across modes.
5. Regression tests:
- bug fixtures map directly to generated module expectations.

## Observability
1. Standard stage timing in generated binaries:
- `load`, `shape_graph_init`, `target_eval`, `validate`, `infer`, `report`
2. Counters:
- generated fast-path hits
- fallback dispatch counts
- per-component violation counts
3. Optional trace tags identifying compiler and backend mode.

## Migration Plan

### Phase 0: Clean-Room Scaffold
1. Add `crates/shacl-srcgen-compiler` with minimal compile CLI.
2. Implement basic `ShapeIR -> SrcGenIR` conversion skeleton.
3. Add deterministic no-op/smoke codegen output and tests.

### Phase 1: Core Source Generation
1. Generate prelude, targets, path evaluators, and run scaffolding.
2. Generate a subset of validators (e.g., class/datatype/cardinality).
3. Add parity tests for subset coverage.

### Phase 2: Broad Validator Coverage
1. Add remaining core components (logical, string, value-range, pair, node/property refs).
2. Add SPARQL/custom components with guarded fallback where necessary.
3. Ensure manifest parity for supported corpus.

### Phase 3: Inference + Reporting Consolidation
1. Generate inference execution code.
2. Consolidate report assembly in generated modules.
3. Add parity/perf gates versus runtime validation and, during migration, the old compiler.

### Phase 4: Optimization and Hybrid Dispatch
1. Introduce compile-time analyses/optimizations in new crate.
2. Add generated fast-path dispatch + shared fallback boundaries.
3. Add explicit counters and benchmark gates per optimization.

### Phase 5: Default Switch
1. Make the srcgen compile path the default operational mode.
2. Remove temporary migration-only compatibility surface once confidence targets are met.

## Acceptance Criteria
1. Deterministic generated output for stable fixtures.
2. Zero semantic regressions on current manifest suites (or documented exceptions).
3. Clear performance story for `specialized` backend.
4. Operational fallback path for unsupported constructs.
5. The compile surface remains stable while the srcgen path finishes rollout.

## Risks and Mitigations
1. Risk: duplicated semantics between generated and runtime paths.
- Mitigation: strict parity tests + shared utility boundaries.
2. Risk: generator complexity growth.
- Mitigation: typed SrcGenIR + modular generators + snapshot tests.
3. Risk: rollout instability.
- Mitigation: phased default switch and temporary migration controls.

## Immediate Next Steps
1. Create `crates/shacl-srcgen-compiler` scaffold.
2. Define initial `SrcGenIR` types and lowering interfaces.
3. Generate minimal runnable module set and add snapshot tests.
4. Wire the srcgen compile path in CLI while preserving a stable migration surface until phase gates are met.
